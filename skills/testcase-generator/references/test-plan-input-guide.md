# 测试计划输入指南

当用户额外提供测试计划时，按本指南处理。

## 适用定位

测试计划作为补充输入使用。

输入分工：

- `PRD / 设计稿 / Figma Make`：回答“要做什么”
- `测试计划`：回答“重点测什么、怎么测、先测什么、不测什么”

## 测试计划可补充的信息

当用户提供测试计划时，优先提取以下内容：

- 测试范围：`In Scope / Out of Scope`
- 模块拆分与优先级：`P0 / P1 / P2`
- 测试策略：功能、回归、兼容性、权限、数据正确性
- 需求覆盖矩阵
- 主流程场景、异常场景、边界场景
- 测试环境与测试数据要求
- Entry / Exit Criteria
- 缺陷分级建议

## 生成测试用例时的使用原则

当测试计划存在时，默认按以下原则处理：

- 业务规则、页面流程、字段约束：仍以 `PRD / 设计稿 / Figma Make` 为主
- 测试优先级、回归范围、冒烟范围、环境准备：以 `测试计划` 为主
- 若测试计划与 PRD 冲突：
  - 先以 PRD 的业务定义为主
  - 同时在输出中标记差异或假设

## 对输出的影响

测试计划输入通常会影响以下输出：

### 1. 完整测试用例

- 用例优先级更明确
- 模块分组更贴近测试执行组织方式
- 异常、边界、权限、回归场景覆盖更完整

### 2. 冒烟集

- 优先按测试计划中的 `P0 / P1` 和主流程场景筛选
- 优先覆盖 Entry / Exit Criteria 中的关键准入项

### 3. UI / Midscene 用例

- 更容易识别应优先自动化的主流程和回归场景
- 可将测试计划中的关键回归路径转成 UI 自动化候选

## 输出说明

使用测试计划作为额外输入时，最终摘要中写明：

- 本次是否使用了测试计划
- 哪些优先级、范围、回归集规则来自测试计划
- 哪些业务规则仍主要来自 PRD

## 常见测试计划结构

如果用户提供的测试计划接近以下结构，应优先参考：

- 文档信息
- 项目背景与测试目标
- 测试范围
- 测试对象拆分与优先级
- 测试策略
- 需求覆盖矩阵
- 测试场景设计
- 测试环境与配置
- 测试数据设计
- 入口准入与出口准出标准
- 缺陷分级建议
