# Apifox CLI + Agent Skills

适用于在支持 Skill 的 Agent 中接入 Apifox CLI，并用自然语言触发接口测试。

## 1. 使用场景

可直接触发的请求示例：

例如：

- `帮我跑一下登录功能的测试，用开发环境`
- `有哪些测试可以执行`
- `用测试环境跑一下支付相关的所有测试`
- `用开发环境和测试环境跑一下登录功能的测试`
- `根据最近的代码变更，在开发环境跑一下受影响的接口测试`

执行分工：

- Apifox CLI 负责真正执行测试
- Agent 负责理解自然语言意图
- Skill 负责定义触发条件、测试选择方式、脚本调用方式和结果解释规则

## 2. 组成方式

核心目录结构如下：

```text
<skills-root>/apifox-tests/
├── SKILL.md
├── env/
│   ├── dev.env
│   ├── test.env
│   └── prod.env
├── scripts/
│   ├── list-tests.js
│   └── run-cli.js
└── tests/
    ├── 支付流程.md
    └── 退款流程.md
```

`<skills-root>` 表示当前 Agent 使用的 skills 根目录。不同 Agent 的默认安装位置可以不同。

## 3. 每个文件的作用

### 3.1 SKILL.md

`SKILL.md` 负责：

- 说明在什么情况下应该启用这个 skill
- 说明启用后按什么流程执行

对 `apifox-tests` 来说，`SKILL.md` 至少要定义：

- 如何选择测试文件
- 如何确认环境
- 何时进入批量执行
- 如何调用 `list-tests.js`
- 如何调用 `run-cli.js`
- 如何解释结果

### 3.2 env/

`env/` 用于存放不同运行环境的变量配置。

示例：

```env
APIFOX_ACCESS_TOKEN=APS-你的访问令牌
APIFOX_ENV_ID=你的环境ID
```

`env/` 的作用：

- 不把 token 直接写进测试定义
- 不用改脚本就能切换 `dev` / `test` / `prod`
- 同一组测试文件可以复用到多个环境

### 3.3 scripts/list-tests.js

这个脚本负责：

- 扫描 `tests/` 目录
- 找到所有 Markdown 测试文件
- 提取首行描述
- 输出“当前有哪些测试可以执行”

适用场景：

- 用户不知道测试文件名，只知道业务模块
- 用户想先看一遍所有可用测试

### 3.4 scripts/run-cli.js

这个脚本负责：

- 读取指定的 Markdown 测试文件
- 提取其中的 Apifox CLI 命令
- 根据传入环境加载对应 `.env`
- 注入 `APIFOX_ACCESS_TOKEN` 和 `APIFOX_ENV_ID`
- 实际执行命令

命令拼装和环境注入由脚本处理。

### 3.5 tests/

`tests/` 目录里的每个 Markdown 文件，表示一个可执行测试。

一个文件通常只需要两部分：

1. 一句简短说明
2. 一段可执行的 Apifox CLI 命令

示例：

````md
> 验证登录、刷新 token、登出等核心接口是否可用。

```bash
apifox run --access-token $APIFOX_ACCESS_TOKEN -t <YOUR_TEST_ID> -e $APIFOX_ENV_ID -n 1 -r html,cli
```
````

文件结构约定：

- 测试说明用于匹配和结果解释
- CLI 命令给脚本做提取和执行
- token 和环境 ID 统一从 `.env` 注入

## 4. 调用方式

把 `apifox-tests` 安装到当前 Agent 的 skills 目录后，可直接用自然语言触发。

随后可以直接用自然语言触发：

### 4.1 执行单个测试

```text
帮我跑一下登录功能的测试，用开发环境。
```

执行流程通常是：

- 找到对应测试文件
- 选择 `dev` 环境
- 执行脚本
- 对结果做总结

### 4.2 查看所有可用测试

```text
有哪些测试可以执行？
```

Agent 应该会先调用 `node scripts/list-tests.js`，把所有测试列出来。

### 4.3 执行某个模块的所有测试

```text
用测试环境跑一下支付相关的所有测试。
```

批量执行时，Skill 里应明确：

- 默认是顺序执行还是先询问
- 是否支持并行
- 批量结果如何汇总

### 4.4 对比不同环境结果

```text
用开发环境和测试环境跑一下登录功能的测试。
```

可用于发现环境配置差异、依赖差异和数据差异。

### 4.5 根据代码变更执行测试

```text
根据最近的代码变更，在开发环境跑一下受影响的接口测试。
```

如果要支持这个能力，还需要在 skill 中约定：

- 如何读取 Git 变更
- 如何从文件路径映射到测试文件
- 找不到映射时如何回退

## 5. 特点

- 测试执行入口从命令变成自然语言
- 环境切换统一通过 `env/` 管理
- 测试定义和执行逻辑分离
- Agent 可以在执行后顺手补结果解释

## 6. 仓库内提供内容

仓库内保留：

- 保留 `SKILL.md`
- 保留 `env/` 模板
- 保留 `list-tests.js` 与 `run-cli.js`
- 保留示例测试文件

接入自己的项目时，通常需要替换：

- `env/*.env`
- `tests/*.md`

如果你的团队需要更复杂的能力，例如：

- 自动按代码变更筛测试
- 多环境结果对比汇总
- 批量执行策略控制
- 报告结构定制

需要更复杂能力时，可在此基础上扩展。
