# Midscene 自然语言 UI 自动化用例标准说明

本文件定义如何将 PRD、设计稿、页面流程说明中的业务场景，整理成**适合 Midscene 消费的自然语言 UI 自动化测试用例**。它与通用测试策略、功能测试标准、自动化候选标准配合使用。

---

## 一、适用场景

- 用户明确提到 `Midscene`、`UI 自动化`、`自然语言步骤`、`根据 PRD/设计稿生成自动化用例`。
- 文档内容以页面流程、交互行为、表单输入、状态反馈、角色操作、页面跳转为主。
- 先产出自然语言测试场景，后续再落为 Midscene 脚本。

---

## 二、推荐输出结构

Midscene 场景下的输出不强制要求固定列名，但应尽量包含以下内容：

- **用例标题**：简短说明场景目标，如「账号密码登录成功后进入工作台」。
- **所属页面/模块**：便于定位场景所在的业务上下文。
- **场景目标/覆盖点**：说明该用例验证的业务规则或关键交互。
- **前置条件**：可选项。仅保留账号状态、环境开关、预置数据、登录态、mock 条件等无法自然写进步骤的外部依赖。
- **测试数据**：输入值、数据组合、边界值、角色信息。
- **自然语言步骤**：按用户可执行的页面操作顺序描述，并优先吸收页面打开、入口进入、可选数据检查、视图切换确认等准备动作。
- **断言要点**：明确页面、文案、控件状态、业务结果应如何变化。
- **自动化建议**：可选，说明是否建议用 Midscene 实现，以及可能的不稳定因素。

---

## 三、写法约束

### 3.1 步骤描述

- 优先使用页面可见元素与业务语义，例如「点击『登录』按钮」「在『手机号』输入框输入 13800000000」。
- 每一步只表达一个动作或一个观察，避免一句话里混入多个动作和多个断言。
- 使用自然语言，不输出 XPath、CSS selector、DOM 层级、坐标、组件内部命名等实现细节。
- 若设计稿中的元素名称不明确，优先使用页面标题、字段标签、按钮文案、卡片标题等稳定锚点来表述。
- 对于「页面已打开」「已选中某条记录」「当前位于某视图」「存在可选项」这类可执行或可观察的前提，优先改写为步骤 1、步骤 2 中的准备动作或首步校验。

### 3.2 断言描述

- 断言必须具体、可观察、可复核，例如：
  - 页面跳转到哪个页面
  - 出现或不出现哪段提示文案
  - 按钮是否可点击
  - 列表是否新增一条记录
  - 表单字段是否回显指定值
  - 当前状态是否变为「已发布」「审核中」等
- 避免使用「显示正常」「提交成功」「结果正确」等模糊表述，除非后面继续补充具体判断依据。

### 3.3 用例拆分

- 正常流、异常流、边界流分别拆成独立用例。
- 一个用例尽量只覆盖一个明确业务目标，避免把大段跨页面、跨角色、跨状态的复杂路径塞进一条用例。
- 若同一页面存在多种关键校验，可按校验类型拆为多条自然语言用例。

### 3.4 前置条件与数据

- Midscene 默认偏好**短前置条件 + 长步骤链**，即尽量把前置条件融入可执行步骤。
- 对依赖测试账号、特定角色、预置数据、环境开关、mock 返回、消息通知的场景，若这些依赖无法通过页面动作自然表达，才保留为前置条件。
- 若前提可以转成动作，优先改写，例如：
  - 不推荐：前置条件「页面已打开，且存在可选数据集」
  - 推荐：步骤 1「打开数据集选择页面」；步骤 2「确认列表中至少存在 1 个可选数据集」
- 对输入类操作，尽量给出明确数据值；若要体现边界或等价类，应写出代表值而不是只写「输入非法值」。
- 设计稿中缺失业务规则时，应明确说明「以下根据常见表单规则补充假设」或类似备注。

## 四、不建议直接转 Midscene 的场景

- 纯视觉审美判断，如是否美观、是否高级、是否舒服。
- 强依赖复杂图像识别、拖拽轨迹、动画细节、画布操作，且页面缺少稳定可见锚点。
- 高度依赖真实第三方能力且无法 mock，例如真实支付、真实短信、外部审批回调。
- 规则尚未稳定、页面频繁变更、设计稿与 PRD 明显不一致的模块。

这些场景可以保留为普通测试用例，或仅标记为「暂不建议 Midscene 自动化」。

---

## 五、与其他标准的配合方式

- **先用通用策略做覆盖**：先按正向、反向、边界值、状态流转等方法完整拆解需求。
- **再用自动化候选标准做筛选**：从完整用例中挑出适合自动化、适合 UI 脚本执行的场景。
- **最后按本标准转成 Midscene 风格**：把筛选出来的 UI 场景整理成自然语言步骤与明确断言。

接口类或纯后端逻辑类用例不必勉强按本标准输出；只有需要页面交互的自动化场景才适合转成 Midscene 风格。

---

## 六、推荐输出模板

当用户要求产出 Midscene 风格自然语言 UI 自动化用例时，可优先按以下模板组织输出：

```md
## 用例 1：<用例标题>

- 所属模块/页面：<模块名或页面名>
- 场景目标：<本用例验证的业务目标或关键交互>
- 用例类型：<正向 / 反向 / 边界 / 状态流转 / 场景>
- 优先级：<P0 / P1 / P2>
- 是否建议 Midscene 自动化：<是 / 否 / 条件性建议>

### 执行前准备（可选，仅保留无法融入步骤的外部条件）

- <外部依赖 1，例如存在测试账号>
- <外部依赖 2，例如 mock 已开启>

### 测试数据

- <字段或数据点 1>：<测试值>
- <字段或数据点 2>：<测试值>

### 执行步骤（含准备动作）

1. <步骤 1，先写打开页面、进入入口、确认可操作条件等准备动作>
2. <步骤 2，说明输入、选择或切换的具体内容>
3. <步骤 3，说明触发动作>
4. <步骤 4，必要时补充等待、切换、再次观察>

### 断言要点

- <断言 1，说明页面应出现的结果>
- <断言 2，说明不应出现的结果>
- <断言 3，说明业务状态或数据变化>

### 备注

- <可选，记录假设、风险点、mock 要求、数据清理方式等>
```

若用户只需要纯场景列表，也可以压缩为更轻量的格式，但仍建议保留以下最小信息：

```md
- 用例标题：
- 步骤：
- 断言：
- 备注（可选）：
```

## 七、完整示例

不推荐：

- 前置条件：存在一个状态正常的已注册账号。
- 步骤 1：打开登录页。
- 步骤 2：输入正确账号密码并登录成功。

推荐：

- 执行前准备：存在一个状态正常的已注册账号。
- 步骤 1：打开登录页。
- 步骤 2：确认页面显示「手机号」和「密码」输入框，以及「登录」按钮。
- 步骤 3：在「手机号」输入框输入 `13800000000`。
- 步骤 4：在「密码」输入框输入 `Test@123456`。
- 步骤 5：点击「登录」按钮。
- 断言：页面跳转到工作台；页面右上角显示当前用户昵称；登录页中的错误提示不出现。

按推荐模板展开后，可写成：

```md
## 用例 1：账号密码登录成功后进入工作台

- 所属模块/页面：登录页
- 场景目标：验证已注册且状态正常的用户可以通过账号密码成功登录
- 用例类型：正向
- 优先级：P0
- 是否建议 Midscene 自动化：是

### 执行前准备（可选，仅保留无法融入步骤的外部条件）

- 系统中存在一个状态正常的已注册账号。
- 当前账号未被锁定，且密码未过期。

### 测试数据

- 手机号：13800000000
- 密码：Test@123456

### 执行步骤（含准备动作）

1. 打开登录页。
2. 确认页面显示「手机号」输入框、「密码」输入框和「登录」按钮。
3. 在「手机号」输入框输入 `13800000000`。
4. 在「密码」输入框输入 `Test@123456`。
5. 点击「登录」按钮。

### 断言要点

- 页面跳转到工作台首页。
- 页面右上角显示当前用户昵称。
- 页面中不出现「手机号或密码错误」提示。

### 备注

- 若测试环境接入单点登录，需要先关闭自动跳转或使用账号密码登录入口。
```
