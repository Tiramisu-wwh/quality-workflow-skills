# PRD 准入评审工具

## 核心功能

根据 PRD 准入检查清单对用户提供的 PRD 文档进行逐项评审，生成符合准入清单格式的 Excel 评审报告。若用户同时提供 Figma Make，则应把它视为 PRD 的补充输入，用于补足页面结构、交互流程、字段和原型实现线索。

## 支持的文档格式

- Word 文档 (.docx)
- PDF 文档 (.pdf)
- Markdown 文档 (.md)
- Figma 设计稿链接
- Figma Make 代码包 (.zip)
- 已解压的 Figma Make 工程目录

可处理单个或多个文档。

## 执行要求（必须遵守）

**必须完整执行所有步骤，不能在评审完成后停止**：

1. 读取检查清单
2. 分析 PRD 文档
3. 逐项评审（输出文本报告）
4. **准备评审结果数据（必须）**
5. **调用 Python 脚本生成 Excel（必须）**
6. **验证 Excel 文件生成成功（必须）**
7. **输出文件路径（必须）**

**关键原则**：
- 不能只输出文本报告就结束
- 必须生成 Excel 文件并确认成功
- 必须报告文件的绝对路径

## 评审结果数据结构

调用 generate_report.py 时，需要准备以下格式的数据：

```python
# 评审结果（必须包含1-44项）
review_results = {
    '1': {'result': '通过', 'note': '需求背景清晰'},
    '2': {'result': '不通过', 'note': '缺少业务价值说明'},
    '3': {'result': '未涉及', 'note': '无历史数据'},
    # ... 继续到第44项
}

# 项目信息（可选）
project_info = {
    'project_name': '从PRD文档中提取',
    'product_manager': '',  # 如果文档中没有则为空
    'version': '从文档中提取',
    'requirement_link': '',
    'expected_release_date': '',
}
```

**注意事项**：
- 所有44项都必须有结果，不能缺失
- 结果值只能是：通过、不通过、部分通过、未涉及
- 备注不能为空字符串，不涉及时为空字符串

## 工作流程

### 步骤 1：加载检查清单

读取 [CHECKLIST.md](references/CHECKLIST.md) 获取完整的检查项清单和评审标准。
若存在 Figma Make 输入，同时参考 [FIGMA_MAKE_INPUT_GUIDE.md](references/FIGMA_MAKE_INPUT_GUIDE.md) 理解联合输入规则。

### 步骤 2：分析 PRD 文档

根据用户提供的文档格式，使用相应的工具读取内容：

- **Word 文档**：使用 `document-skills:docx` 技能读取
- **PDF 文档**：使用 `document-skills:pdf` 技能读取
- **Markdown 文档**：使用 Read 工具读取
- **Figma 设计稿**：使用 `mcp__figma__get_figma_data` 获取设计内容
- **Figma Make**：优先读取 `README.md`、`src/app/App.*`、页面目录、`api`、`schema`、`types` 等关键文件；如果输入是 `.zip`，先解压再分析

对于多个文档，逐个读取并合并分析。

若用户同时提供 `PRD + Figma Make`，默认采用以下原则：

- `PRD` 负责业务规则、范围、验收标准
- `Figma Make` 负责页面、交互、字段、状态和原型实现线索
- 若两者冲突，要在评审备注中明确记录差异，不要静默忽略

### 步骤 3：逐项评审

对照检查清单中的每一项，在 PRD 文档中查找相关内容：

- **通过**：文档中包含完整、明确的相关内容
- **不通过**：文档中缺少或内容不完整
- **部分通过**：内容存在但不完整，在"备注"栏说明缺失部分

**评审原则**：
- 只检查清单中列出的内容，不额外添加检查项
- 如果某项内容分散在多个文档中，只要能找到完整信息即为通过
- 对于"界面原型/设计稿已提供"等项，Figma 链接即视为通过
- 备注栏用于记录具体缺失内容或问题描述

### 步骤 4：生成 Excel 报告（必须执行）

**这是关键步骤，不能跳过或仅输出文本报告**。

#### 4.1 准备数据

将评审结果转换为脚本所需的数据结构，确保：
- 包含所有44项检查结果
- 每项都有 result 和 note 字段
- 数据格式符合 Python 脚本要求

#### 4.2 调用脚本

创建临时 Python 脚本传入数据：
```bash
python3 << 'EOF'
import sys
import os

# 获取技能目录的相对路径
skill_dir = os.path.dirname(os.path.abspath(__file__))
scripts_dir = os.path.join(skill_dir, 'scripts')
sys.path.insert(0, scripts_dir)

from generate_report import generate_review_report

# 评审结果（44项）
review_results = {
    '1': {'result': '通过', 'note': '需求背景清晰'},
    '2': {'result': '不通过', 'note': '缺少业务价值说明'},
    '3': {'result': '未涉及', 'note': '无历史数据'},
    # ... 继续到第44项
}

# 项目信息
project_info = {
    'project_name': '从PRD文档中提取',
    'product_manager': '',
    'version': '从文档中提取',
    'requirement_link': '',
    'expected_release_date': '',
}

# 生成报告（使用当前工作目录）
output_path = os.path.join(os.getcwd(), 'PRD准入评审报告.xlsx')
generate_review_report('', output_path, review_results, project_info)
print(f"✅ Excel报告已生成: {output_path}")
EOF
```

#### 4.3 验证生成

如果文件不存在或大小异常（< 5KB），说明生成失败，需要重试。

### 步骤 5：输出报告（必须包含文本和Excel）

#### 5.1 文本报告输出

在对话中输出以下内容：
- 三个类别的详细评审结果（通过率、通过项数）
- 问题清单（按严重程度排序）
- 最终评审结论

#### 5.2 Excel 报告输出（必须）

**必须同时输出 Excel 文件路径**，包括：
- 文件的绝对路径
- 文件大小
- 生成时间
- 生成成功确认

#### 输出格式示例：

```
## PRD 准入评审结果

### 文本报告
[完整的文本评审结果...]

### Excel 报告
- 文件路径: {当前工作目录}/PRD准入评审报告.xlsx
- 文件大小: 12.5 KB
- 生成时间: 2026-03-13 09:45:00

✅ Excel 报告已成功生成，请查看文件。
```

## 通过标准

### 类别通过标准

1. **文档完整性检查**：通过率 ≥ 80%（25项中至少20项通过）
   - 核心项（需求背景、功能列表、验收标准、依赖系统）必须全部通过

2. **质量标准检查**：通过率 ≥ 75%（16项中至少12项通过）

3. **其他相关资料**：通过率 ≥ 67%（3项中至少2项通过）

### 最终评审结论

根据三个类别的通过率综合判定：

- **通过**：三个类别均达到上述标准
- **有条件通过**：接近标准但有个别遗漏（通过率 ≥ 65%且核心项基本满足），可在补充相关内容后通过
- **不通过**：任一类别未达到最低要求或核心项严重缺失

## Excel 报告格式

使用 [准入清单模板](assets/PRD准入检查清单模板.xlsx) 作为基准，保持：
- 所有列名和结构完全一致
- 格式和样式与模板相同
- 问题清单按严重程度排序（高、中、低）

## 特殊处理

### 数据变更影响（1.7 节）

- **检查项 22-25**（数据变更说明、数据迁移方案、数据回滚方案、历史数据处理）
- 若有历史数据影响且说明完整则为"通过"
- 若无历史数据影响则标记为"未涉及"，而非"不通过"
- 备注栏：有内容时记录具体方案，无内容时为空

### 依赖系统/接口（1.6 节）

- **检查项 20-21**（依赖系统已列出、第三方服务依赖已说明）
- 若有依赖关系且说明完整则为"通过"
- 若无依赖关系则标记为"未涉及"，而非"不通过"
- 备注栏：有内容时记录具体依赖，无内容时为空

### 异常场景和错误处理

- **检查项 36-37**（正常场景已覆盖、边界条件已定义、异常场景已覆盖、错误处理已说明）
- **正常场景和边界条件**（36）：检查是否覆盖了正常流程和边界条件
- **异常场景和错误处理**（37）：主要指用户操作友好上的异常和错误处理，不包括网络故障、系统崩溃等不可控因素，在用户操作不当和程序执行失败时是否有给出友好引导提示

### Figma 设计稿

- 合并所有文档内容进行分析
- 不同文档中同一内容不重复计算
- 在问题清单中标注缺失内容的文档来源

### Figma Make 联合输入

- Figma Make 按可运行的原型工程处理，不按普通设计链接处理
- 可从中补充识别页面清单、菜单层级、弹窗、表单字段、列表列、版本状态、接口与数据对象线索
- 当 PRD 中缺少页面级细节时，可使用 Figma Make 作为补充证据
- 但涉及业务目标、规则边界、验收口径时，仍以 PRD 为主

## 注意事项

- 评审过程中保持客观，只基于清单内容判断
- 对于模糊或不确定的内容，倾向于不通过并说明原因
- 备注栏尽量详细，帮助用户理解问题所在
- 问题清单按类别组织，便于后续修改
