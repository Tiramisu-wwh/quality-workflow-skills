# Figma Make 输入指南

## 适用范围

本指南适用于所有“以 PRD 为主输入”的质量与测试类 skill。

`PRD + Figma Make` 默认作为联合输入使用：

- `PRD` 负责提供业务目标、范围、规则、角色、约束、验收标准
- `Figma Make` 负责提供页面结构、交互流程、字段命名、状态变化、前后端接口与数据模型线索

## 支持的 Figma Make 输入形式

以下输入均视为有效的 Figma Make 输入：

- Figma Make 导出的代码包 `.zip`
- 已解压的 Figma Make 工程目录
- Figma Make 相关的 README、页面代码、接口代码、schema 文件集合

## 读取顺序

当用户同时提供 PRD 与 Figma Make 时，按以下顺序处理：

1. 先读 PRD，明确业务目标、角色、流程、规则、验收口径
2. 再读 Figma Make，补充页面结构、交互细节、字段、状态、接口和数据模型线索
3. 最后合并形成统一理解，识别差异、缺口和假设

## Figma Make 重点关注内容

优先关注以下文件或目录：

- `README.md`
- `src/app/App.*`、`src/main.*`
- `routes`、`pages`、`components/pages`
- `api`、`services`、`controllers`
- `schema`、`types`、`constants`

重点提取的信息包括：

- 页面与模块清单
- 菜单、路由、弹窗、抽屉、表单、表格
- 字段名、校验规则、状态切换
- 接口名、请求参数、返回结构
- 数据对象、版本对象、分组对象等模型信息

## 联合输入判定原则

当 PRD 与 Figma Make 同时存在时，默认遵循以下原则：

- 业务规则、需求范围、验收标准：以 `PRD` 为主
- 页面交互、字段展示、组件结构、状态呈现：以 `Figma Make` 为主
- 若两者冲突：
  - 优先记录差异，不要静默覆盖
  - 测试设计与门禁评审时应明确写出“以 PRD 为准”或“以现有原型实现为准”的假设

## 对输出的补充

### 对 PRD/安全准入类 skill

补充关注：

- 页面原型是否存在
- 页面流程是否能闭环
- 字段、状态、交互是否明确
- 是否已经体现出接口、数据对象和异常提示

### 对测试用例类 skill

补充关注：

- 页面元素与交互动作
- UI 自动化候选场景
- 列表、详情、创建、编辑、删除等典型页面流
- 接口与数据模型线索

## 输出说明

使用 `PRD + Figma Make` 联合输入时，最终输出中写明：

- 本次分析是否同时使用了 PRD 与 Figma Make
- 哪些结论主要来自 PRD
- 哪些结论主要来自 Figma Make
- 哪些地方存在差异、缺失或需要人工确认
